# ESC Trading Guide

**Document Reference:** WR-UG-002 | **Version:** 1.0 | **Date:** 5 April 2026
**Purpose:** Step-by-step walkthrough of an Energy Savings Certificate trade on the WREI Platform
**Audience:** ESC brokers, obligated entities, accredited certificate providers

---

## 1. Accessing the Trading Interface

1.1. Navigate to `/trade` from any page using the Trading (TRD) tab in the Bloomberg-style navigation bar.

1.2. The trade page loads with a three-panel layout managed by `app/trade/page.tsx`. The page is a client component (`'use client'`) that coordinates state through custom hooks (`useTradeState`, `useTradeAPI`, `useTradeActions`, `useTradeHistory`).

1.3. At the top of the page, the `TradeModeSelector` component offers three interface modes.

1.3.1. **Standard** — The default trading interface with instrument selection, order book, AI negotiation chat, and trade blotter.

1.3.2. **Bulk** — The `BulkNegotiationDashboard` for large-volume purchases across multiple counterparties (see Scenario E in the user scenarios document).

1.3.3. **Professional** — The `ProfessionalInterface` with investor profile, portfolio allocation, and institutional-grade decision tools.

1.4. For a standard ESC trade, remain in Standard mode.

---

## 2. Selecting ESC as the Instrument

2.1. In the left panel, the `InstrumentSwitcher` component displays all eight tradeable instruments grouped into three category tabs.

2.1.1. **Certificates (CRT)** — ESC, VEEC, PRC, ACCU, LGC, STC.

2.1.2. **Carbon Tokens (CTK)** — WREI-CC.

2.1.3. **Asset Tokens (ATK)** — WREI-ACO.

2.2. Click the **Certificates** tab if not already selected, then click **ESC — Energy Savings Certificate**.

2.3. When you select ESC, the `onInstrumentChange` handler fires, which resolves the current ESC pricing via the pricing engine (`resolveInstrumentPricing`) and updates the interface. The order book, price chart, and negotiation parameters all reconfigure for ESC.

2.4. The current ESC spot price is displayed alongside the instrument name. Each price carries a source label — "Live," "Cached," or "Indicative" — corresponding to the data feed tier (live scrape within 24 hours, stale but real data, or simulated data respectively).

---

## 3. Reviewing Market Data

3.1. With ESC selected, the left panel displays market context through two components.

### 3.1.1. Order Book Panel

The `OrderBookPanel` component (`components/trading/OrderBookPanel.tsx`) shows simulated bid/ask depth at multiple price levels for the selected instrument. It receives the instrument type and the current spot price as props.

The order book displays the following information.

- Bid side (buy orders) with volume at each price level.
- Ask side (sell orders) with volume at each price level.
- Spread between best bid and best ask.
- Total depth on each side.

For ESC, typical bid/ask levels cluster around the A$47.80 spot reference, with depth reflecting the simulated counterparty inventory defined in `lib/demo/seed-data.ts`.

### 3.1.2. Token Detail Panel

The `TokenDetailPanel` component (`components/trading/TokenDetailPanel.tsx`) appears only when WREI-CC or WREI-ACO is selected. For ESC trades, this panel is not displayed.

### 3.1.3. Price Context

The ESC pricing configuration in `lib/negotiation-config.ts` provides the following market references.

- Current NSW ESC spot: A$47.80 (from AEMO trading data).
- ESC forward curve (12-month): A$52.15.
- Historical volatility range: A$38-68 (12-month).
- Data sources: AEMO, NSW ESC Registry, IPART, Clean Energy Regulator.
- Settlement cycle: T+2 (registry transfer).
- Minimum trade size: 1,000 ESCs.

**Note:** In demo mode, price history is generated by `generatePriceHistory('ESC', 23.00, 0.02)` in the seed data module, producing 30 days of bounded random walk data. In production, prices are scraped from Ecovantage and Northmore Gordon via the feed manager's adapter system.

## 4. Choosing a Trade Method

4.1. Below the instrument switcher, select the trade type via the interface controls. ESC supports two primary methods.

### 4.1.1. Direct Order

Place a market or limit order at the current spot price. The order executes immediately against the simulated order book. Suitable for straightforward purchases where price discovery is not required.

### 4.1.2. AI Negotiation

Engage the Claude-powered negotiation agent to negotiate the best price on your behalf. The agent conducts a multi-turn conversation, reasoning about market conditions, adapting its strategy, and working toward a deal. This is the recommended method for any trade where price optimisation matters.

4.2. To proceed with AI Negotiation, you must first configure two additional parameters.

---

## 5. Configuring AI Negotiation Parameters

5.1. The `TokenTypeSelector` component lets you choose the token type context. For ESC trades, the instrument is already set — this selector primarily differentiates between carbon credits, asset co tokens, and dual portfolio modes for WREI-native instruments. It is disabled once trading has started.

5.2. The `PersonaSelector` component (`app/trade/_components/PersonaSelector.tsx`) determines the buyer profile the AI agent will negotiate against. Available ESC-specific personas are defined in `lib/trading/personas/esc-personas.ts` and include the following.

5.2.1. **NSW ESC Obligated Entity Buyer** — Procurement officer at an electricity retailer. Highly price-sensitive, deadline-driven, references the penalty rate as their BATNA.

5.2.2. **ESC Trading Desk (Institutional)** — Environmental markets trader at an institutional desk. Sophisticated, demands tight spreads and fast execution.

5.2.3. **Government Energy Efficiency Buyer** — Procurement officer at a NSW government agency. Process-oriented, requires formal documentation.

5.2.4. **ESC Accredited Certificate Provider (Seller)** — An ACP seeking to sell registered ESCs. Inventory-constrained, cash-flow sensitive.

5.3. General personas (Corporate Compliance Officer, ESG Fund Portfolio Manager, Carbon Trading Desk Analyst, Sustainability Director, Government Procurement Officer) are also available and work with ESC as the instrument.

5.4. Select a persona, then click **Start AI Negotiation**. The `handleStartTrading` function initialises the negotiation session, sets the anchor price based on the instrument's pricing configuration, and sends the opening message to the Claude API via the server-side route at `/api/negotiate/route.ts`.

---

## 6. Monitoring the Negotiation

6.1. Once the negotiation begins, the right panel displays the `ChatInterface` component — a real-time conversation between you and the AI negotiation agent.

6.2. Type messages in the input field and press Enter (or click Send). Each message passes through the defence layer (`lib/defence.ts`), which sanitises the input before forwarding it to the Claude API. The following threat categories are detected and neutralised.

6.2.1. Role override attempts (e.g., "you are now," "ignore previous instructions").

6.2.2. Strategy extraction attempts (e.g., "what is your minimum," "reveal your system prompt").

6.2.3. Format manipulation attempts (e.g., "respond in this format," "output as JSON").

6.3. The `NegotiationDashboard` component in the left panel provides real-time analytics alongside the conversation. It displays the following.

6.3.1. **Round counter** — Current negotiation round.

6.3.2. **Phase indicator** — Opening, Elicitation, Negotiation, Closure, or Escalation.

6.3.3. **Agent's offer** — Current price, displayed as `AUD $XX.XX/ESC` for ESC instruments.

6.3.4. **Your anchor** — The buyer's stated price anchor for reference.

6.3.5. **Concession tracking** — Visual progress bar showing how much of the maximum 20% concession has been used.

6.3.6. **Argument classification** — The system classifies each buyer message (price challenge, fairness appeal, time pressure, information request, relationship signal, authority constraint, emotional expression, or general).

6.3.7. **Emotional state** — The system detects the buyer's emotional state (frustrated, enthusiastic, sceptical, satisfied, neutral, or pressured).

6.3.8. **Threat level** — None, low, medium, or high, based on defence layer analysis.

6.4. The `AnalyticsInlinePanel` component appears below the chat, providing additional analytical context during the negotiation.

---

## 7. Reviewing and Confirming the Trade

7.1. When the negotiation reaches agreement, the agent moves to the Closure phase. The agreed terms are summarised in the chat interface.

7.2. You can end the negotiation at any time using the controls in the `ChatInterface`. The available actions are as follows.

7.2.1. **End Trading** — Conclude the session and finalise any agreed terms.

7.2.2. **Request Human** — Escalate to a human WREI representative for complex deal structuring.

7.2.3. **Reset Trading** — Clear the session and start fresh.

7.3. Upon confirmation, a trade record is created and the `TradeModals` component may display a provenance certificate (via `ProvenanceCertificate` from `components/trading/ProvenanceCertificate.tsx`) and a trading scorecard.

---

## 8. Settlement Process — TESSA Transfer Workflow

8.1. ESC settlement follows the NSW registry transfer process through TESSA (Transfer and Surrender of ESCs Application), administered by IPART. The settlement lifecycle is tracked in the `settlements` table with the following states.

8.1.1. **Pending** — Trade confirmed, awaiting transfer initiation.

8.1.2. **Processing** — TESSA transfer initiated. The seller's ESCs are being transferred to the buyer's registry account.

8.1.3. **Completed** — Transfer recorded in TESSA. Settlement complete.

8.1.4. **Failed** — Transfer failed. Requires manual intervention.

8.2. The estimated settlement cycle for ESC is T+2 (two business days for registry transfer).

8.3. Current TESSA integration is manual confirmation — the platform tracks the workflow stages, but the actual registry transfer is initiated outside the platform. Future API integration is planned when IPART makes TESSA API access available.

8.4. Settlement method for ESC is recorded as `'registry'` in the database, distinguishing it from `'zconnect'` (on-chain) used for WREI tokens and `'manual'` for offline arrangements.

## 9. Viewing the Trade in the Blotter

9.1. The `TradeBlotter` component (`components/trading/TradeBlotter.tsx`) appears at the bottom of the trade page, below the main trading panels.

9.2. The blotter displays all trades with the following columns.

9.2.1. **Trade ID** — Unique identifier (e.g., TRD-DEMO-001).

9.2.2. **Instrument** — The certificate type (ESC in this case).

9.2.3. **Direction** — Buy or sell.

9.2.4. **Quantity** — Number of certificates.

9.2.5. **Price** — Price per unit in the instrument's currency.

9.2.6. **Total Value** — Quantity multiplied by price.

9.2.7. **Status** — Pending, confirmed, settled, or cancelled.

9.2.8. **Executed At** — Timestamp of trade execution.

9.2.9. **Settled At** — Timestamp of settlement completion (null if not yet settled).

9.3. The blotter supports sorting by any column (ascending/descending) and filtering by instrument type. The `instrumentFilter` prop is set to the currently selected instrument, so it shows ESC trades by default when ESC is the active instrument.

9.4. Export functionality is available via the report generator (`lib/trading/compliance/report-generator.ts`), which produces CSV output suitable for compliance reporting and internal audit.

9.5. In demo mode, three pre-seeded trades appear in the blotter, showing settlement progression at different stages.

9.5.1. TRD-DEMO-001 — 25,000 ESC buy at A$22.80, status: settled (3 days ago).

9.5.2. TRD-DEMO-002 — 5,000 VEEC buy at A$83.20, status: confirmed (1 day ago).

9.5.3. TRD-DEMO-003 — 50,000 ESC buy at A$23.10, status: pending (2 hours ago).

## 10. Status Bar

10.1. The `TradingStatusBar` component at the bottom of the interface displays a persistent summary of the active negotiation state, providing at-a-glance visibility of the current session without scrolling.

## 11. Notes on Live vs. Simulated Data

The following table clarifies which data elements are live and which are simulated in the current platform version.

| Element | Demo Mode | Production Mode |
|---|---|---|
| ESC spot price | Simulated (bounded random walk from A$23.00 base) | Live scrape from Ecovantage / Northmore Gordon |
| Order book | Simulated (generated depth) | Simulated (no live CLOB available) |
| Counterparties | 5 pre-seeded (EfficientAus, GreenCert NSW, ESC Direct, CarbonTrade, National Energy Services) | Real counterparty accounts |
| AI negotiation | Live Claude API call | Live Claude API call |
| Trade blotter | Pre-seeded demo trades + session trades | Persisted to Vercel Postgres |
| Settlement | Simulated status progression | Manual TESSA confirmation (future: API) |
| Audit trail | Logged to database | Logged to database |
